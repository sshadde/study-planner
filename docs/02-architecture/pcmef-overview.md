# Описание архитектуры PCMEF

## Общая архитектурная идея

Система проектируется как клиент-серверное мобильное приложение. Мобильный клиент отвечает за пользовательский интерфейс, состояние экранов, сетевой обмен и локальный кэш. Серверная часть реализует защищенный REST API, бизнес-логику, работу с PostgreSQL и документацию OpenAPI.

Архитектурной основой проекта является PCMEF:

| Слой | Название | Ответственность в проекте |
|---|---|---|
| P | Presentation | UI мобильного приложения на Jetpack Compose, экраны, компоненты, отображение состояний |
| C | Control | REST-контроллеры Spring Boot, обработка HTTP-запросов, DTO, валидация входных данных |
| M | Mediator | Сервисы бизнес-логики, транзакции, проверка прав, координация сценариев |
| E | Entity | Доменные и JPA-сущности: User, Assignment, Course, Reminder |
| F | Foundation | Репозитории, доступ к PostgreSQL, мапперы, инфраструктурные адаптеры |

## Адаптация PCMEF для мобильной траектории

В мобильной траектории Presentation физически находится в Android-приложении, а Control, Mediator, Entity и Foundation реализуются на сервере. Взаимодействие между мобильным клиентом и сервером выполняется через REST API.

Дополнительно мобильный клиент содержит собственные технические слои:

| Модуль клиента | Назначение |
|---|---|
| presentation | Compose-экраны, навигация, UI-компоненты |
| state | ViewModel, UI state, обработка пользовательских событий |
| remote | Retrofit API, сетевые DTO |
| local | Room DAO, локальные сущности, кэш |
| sync | Синхронизация локальных и серверных данных |

Эти модули не заменяют серверный PCMEF, а обеспечивают работу Presentation-слоя в мобильной среде.

## Правило зависимостей

Зависимости направлены сверху вниз:

```text
Presentation -> Control -> Mediator -> Entity -> Foundation
```

Практическая интерпретация для backend:

```text
controller -> service -> entity / repository interface -> repository implementation -> database
```

Слои не должны создавать обратные зависимости. Например, Entity не знает о REST-контроллерах, Foundation не содержит бизнес-правила, а Control не обращается к базе данных напрямую.

## Распределение ответственности

| Слой | Разрешено | Запрещено |
|---|---|---|
| Presentation | Отображать данные, вызывать API, показывать loading/error/empty states | Выполнять серверную бизнес-логику |
| Control | Принимать HTTP-запросы, валидировать DTO, возвращать статусы ответа | Обращаться к PostgreSQL напрямую |
| Mediator | Выполнять бизнес-правила, транзакции, проверку владения данными | Знать о Compose-экранах или Android API |
| Entity | Хранить состояние и доменные методы | Делать SQL-запросы или HTTP-вызовы |
| Foundation | Выполнять доступ к данным и маппинг | Решать, можно ли студенту выполнить бизнес-действие |

## Ключевые архитектурные свойства

- Разделение пользовательского интерфейса и серверной бизнес-логики.
- REST API как контракт между мобильным клиентом и сервером.
- JWT-аутентификация для защищенных сценариев.
- PostgreSQL как основное хранилище актуальных данных.
- Room-кэш на мобильном устройстве для оффлайн-доступа.
- Тестируемость сервисного слоя через интерфейсы репозиториев.
- Возможность расширения: push-уведомления, интеграция с LMS, веб-админка.
