# ADR: архитектурные решения

## ADR-001. Выбор мобильной траектории

**Статус:** принято.

**Контекст:** тема проекта предполагает мобильное приложение для студентов.

**Решение:** использовать траекторию В: Android Native клиент и Spring Boot backend.

**Последствия:**

- проект соответствует заявленной теме;
- требуется реализовать минимум 5 экранов, REST API, JWT и оффлайн-кэш;
- увеличивается объем работ по сравнению с десктопной траекторией, но повышается итоговая ценность проекта.

## ADR-002. Использование PCMEF как основной архитектуры

**Статус:** принято.

**Контекст:** методические указания требуют проектирования на основе PCMEF.

**Решение:** разделить backend на `control`, `mediator`, `entity`, `foundation`, а мобильный клиент рассматривать как Presentation-слой с собственными техническими модулями `presentation`, `state`, `remote`, `local`, `sync`.

**Последствия:**

- архитектура соответствует требованиям курсового проекта;
- бизнес-логика концентрируется в сервисах;
- контроллеры и репозитории остаются тонкими и тестируемыми.

## ADR-003. REST API как контракт между клиентом и сервером

**Статус:** принято.

**Контекст:** мобильный клиент должен взаимодействовать с серверной частью. Методические указания требуют REST API и OpenAPI/Swagger.

**Решение:** реализовать REST API с JSON DTO и документацией Swagger UI.

**Последствия:**

- мобильный клиент не зависит от внутренней структуры backend;
- API можно тестировать отдельно через Swagger/Postman;
- появляется основа для будущей веб-админки или интеграции с внешними системами.

## ADR-004. JWT-аутентификация

**Статус:** принято.

**Контекст:** мобильная траектория требует JWT и разграничение доступа по ролям.

**Решение:** использовать Spring Security, BCrypt для паролей и JWT access token для защищенных эндпоинтов.

**Последствия:**

- сервер не хранит пользовательскую сессию;
- мобильный клиент должен безопасно хранить токен;
- требуется обработка истечения токена и ошибок `401 Unauthorized`.

## ADR-005. PostgreSQL как серверная база данных

**Статус:** принято.

**Контекст:** методические указания для мобильной траектории рекомендуют PostgreSQL и Spring Data JPA.

**Решение:** использовать PostgreSQL как основное хранилище актуальных данных, Spring Data JPA для ORM и DDL/миграции для схемы данных.

**Последствия:**

- система получает надежное реляционное хранение;
- можно явно описать ER-модель, ключи, индексы и ограничения;
- для локального запуска потребуется настроенная БД или Docker Compose.

## ADR-006. Room для локального кэша Android

**Статус:** принято.

**Контекст:** мобильная траектория требует локальное кэширование и оффлайн-режим.

**Решение:** использовать Room для хранения кэшированных заданий, дисциплин и статусов на устройстве.

**Последствия:**

- пользователь может просматривать данные без сети;
- требуется маппинг между сетевыми DTO, локальными моделями и UI-моделями;
