# Спецификация прецедентов

## UC-02. Войти в систему

| Поле | Описание |
|---|---|
| Цель | Предоставить пользователю доступ к защищенным функциям приложения |
| Основной актор | Гость |
| Заинтересованные стороны | Студент |
| Предусловия | У пользователя есть зарегистрированная учетная запись |
| Постусловия | Пользователь получает JWT-токен, мобильное приложение сохраняет состояние авторизации |
| Основной сценарий | 1. Пользователь открывает экран входа. 2. Вводит email и пароль. 3. Мобильное приложение отправляет запрос `POST /api/auth/login`. 4. Сервер проверяет учетные данные. 5. Сервер возвращает access token и данные пользователя. 6. Приложение сохраняет токен в безопасном хранилище. 7. Пользователь переходит на главный экран заданий. |
| Альтернативный сценарий A1 | Если email или пароль неверны, сервер возвращает ошибку `401 Unauthorized`, приложение показывает сообщение об ошибке |
| Альтернативный сценарий A2 | Если сеть недоступна, приложение показывает сообщение о невозможности входа и предлагает повторить попытку |
| Исключения | Заблокированная учетная запись, невалидный формат email, пустой пароль |
| Приоритет | Высокий |

## UC-05. Создать задание

| Поле | Описание |
|---|---|
| Цель | Позволить студенту добавить новое учебное задание |
| Основной актор | Студент |
| Заинтересованные стороны | Студент |
| Предусловия | Пользователь авторизован; существует хотя бы одна дисциплина или пользователь может создать ее при добавлении задания |
| Постусловия | Новое задание сохранено на сервере и отображается в списке заданий; локальный кэш обновлен |
| Основной сценарий | 1. Студент открывает экран создания задания. 2. Заполняет название, описание, дисциплину, дедлайн, приоритет и начальный статус. 3. При необходимости добавляет напоминание. 4. Приложение проверяет обязательные поля. 5. Приложение отправляет запрос `POST /api/assignments`. 6. Сервер проверяет JWT-токен и валидирует данные. 7. Сервер сохраняет задание. 8. Приложение обновляет список заданий и локальный кэш. |
| Альтернативный сценарий A1 | Если обязательные поля не заполнены, приложение подсвечивает ошибки и не отправляет запрос |
| Альтернативный сценарий A2 | Если сеть недоступна, приложение сохраняет черновик локально и помечает его как ожидающий синхронизации |
| Исключения | Некорректная дата дедлайна, отсутствующая дисциплина, истекший JWT-токен |
| Приоритет | Высокий |

## UC-09. Искать и фильтровать задания

| Поле | Описание |
|---|---|
| Цель | Быстро находить задания по заданным критериям |
| Основной актор | Студент |
| Заинтересованные стороны | Студент |
| Предусловия | Пользователь авторизован; в системе есть задания или локально сохраненный кэш |
| Постусловия | Пользователь видит список заданий, соответствующих фильтрам |
| Основной сценарий | 1. Студент открывает список заданий. 2. Вводит поисковый запрос или выбирает фильтры. 3. Приложение применяет фильтры по дисциплине, статусу и приоритету. 4. При наличии сети приложение запрашивает `GET /api/assignments` с query-параметрами. 5. На экране отображается отфильтрованный список. |
| Альтернативный сценарий A1 | Если сеть недоступна, поиск выполняется по локальному кэшу |
| Альтернативный сценарий A2 | Если ничего не найдено, приложение показывает пустое состояние |
| Исключения | Невалидные параметры фильтра, ошибка сервера |
| Приоритет | Средний |
